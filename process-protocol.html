The Shape

Every dizzy process — procedure or policy — is just a function with two arguments: an input and a context. The context carries an optional emit object whose keys are named callback functions. That's the entire contract.
typescript process protocol 
// The universal Process type
type Process<
  Input,
  Ctx extends { emit?: Record<string, (...args: any[]) => void> }
> = (input: Input, ctx: Ctx) => Promise<void> | void

//  ┌── 1. structured input ──────────────────────────────────────┐
//  │                                                              │
//  │   (input: Input,   ctx: Ctx) => Promise<void> | void        │
//  │                          │                                   │
//  │                          └── 2. context with emit callbacks  │
//  └──────────────────────────────────────────────────────────────┘

// The emit object is the "transport hole":
//   - in-memory  → ctx.emit.foo(payload)  is a direct function call
//   - over AMQP  → ctx.emit.foo(payload)  publishes to a queue topic
//   - over HTTP  → ctx.emit.foo(payload)  POSTs to an endpoint
//
// The process itself never knows which transport is in use.

Procedure shape

Input is a command. Outputs are events via emit. Side-effecting work happens inside. Think: scan files, upload a blob, filter a manifest.

Policy shape

Input is an event. Outputs are commands via emit. Pure coordination logic — decides what to do next. Think: "scan complete → create manifest".

Procedure — partition_scan

A procedure handles a command and emits events. The emit callbacks are injected by the runtime — the function body never touches a queue, socket, or HTTP client.
typescript cmd: start_scan → emits 3 events 
import type { Process } from 'dizzy'

interface StartScanInput {
  rootPath: string            // ← the command payload
}

interface ScanEmit {
  scan_item_found  : (item: { path: string; sha256: string }) => void
  scan_complete    : () => void
  scan_item_failed : (err: { path: string; reason: string }) => void
}

export const partitionScan: Process<StartScanInput, { emit: ScanEmit }> =
  async (input, ctx) => {
    for await (const entry of walkDir(input.rootPath)) {
      try {
        const sha256 = await hashFile(entry.path)
        ctx.emit.scan_item_found({ path: entry.path, sha256 })   // ← emit event
      } catch (err) {
        ctx.emit.scan_item_failed({ path: entry.path, reason: String(err) })
      }
    }
    ctx.emit.scan_complete()                                       // ← emit event
  }

// walkDir and hashFile are ordinary utility functions.
// No queue import. No transport knowledge. Just logic.

Policy — create_image_priority_manifest

A policy reacts to an event and emits commands. It contains no side-effecting work — only the decision of what should happen next. Swap the transport and the logic is identical.
typescript evt: scan_complete → emits 1 command 
import type { Process } from 'dizzy'

interface ScanCompleteEvent {
  scannedItems: Array<{ path: string; sha256: string }>
}

interface ManifestPolicyEmit {
  create_image_priority_manifest: (payload: {
    items: Array<{ path: string; sha256: string }>
  }) => void
}

export const createImagePriorityManifestPolicy: Process<
  ScanCompleteEvent,
  { emit: ManifestPolicyEmit }
> = (input, ctx) => {
  // Policy logic: decide to issue a command based on the event
  if (input.scannedItems.length > 0) {
    ctx.emit.create_image_priority_manifest({ items: input.scannedItems })
  }
  // That's all. No file I/O. No HTTP. No AMQP. Pure coordination.
}

Portability — Same Process, Any Transport

The process function body is identical in all three cases below. Only the wiring changes — the construction of the ctx.emit object that is passed in by the runtime.

In-Memory

Direct function calls. Used in tests, monoliths, or co-located processes.

const ctx = {
  emit: {
    scan_item_found: (item) =>
      updateManifest(item),
    scan_complete: () =>
      triggerNextStep(),
  }
}
await partitionScan(cmd, ctx)

AMQP Queue

Each emit callback publishes a message. Used across separate containers.

const ctx = {
  emit: {
    scan_item_found: (item) =>
      ch.publish(
        'events',
        'scan_item_found',
        Buffer.from(JSON.stringify(item))
      ),
    scan_complete: () =>
      ch.publish('events',
        'scan_complete', Buffer.from('{}')
      ),
  }
}
await partitionScan(cmd, ctx)

HTTP Webhook

Each emit callback POSTs to an endpoint. Used for external integrations.

const ctx = {
  emit: {
    scan_item_found: (item) =>
      fetch(WEBHOOK_URL, {
        method: 'POST',
        body: JSON.stringify({
          event: 'scan_item_found',
          payload: item
        })
      }),
    scan_complete: () =>
      fetch(WEBHOOK_URL, { method: 'POST',
        body: JSON.stringify({
          event: 'scan_complete' }) }),
  }
}
await partitionScan(cmd, ctx)
The transport becomes a deployment-time decision, not a code-time decision. This is why the Process Protocol enables automated IaC generation.

The feat file — scan_and_upload.feat.yaml

A feature is declared as a YAML file listing commands, events, procedures, and policies with their relationships. This is the input to the entire deployment pipeline.
yaml 3 commands 5 events 3 procedures 3 policies 
feature: scan_and_upload

commands:
  start_scan:                        "Initiates a scan to discover files"
  create_image_priority_manifest:    "Creates a prioritized manifest for upload"
  upload_blob_using_manifest:        "Uploads a blob using manifest information"

events:
  scan_item_found:         "Found a file (not folder) while scanning"
  scan_complete:           "Finished scanning"
  scan_item_failed:        "Something went wrong reading a single item"
  priority_manifest_created: "Prioritized manifest successfully created"
  manifest_upload_started:   "Upload has started for a manifest item"

procedures:
  partition_scan:
    description: "Recursively scans directory, hashes each file with SHA256"
    command: start_scan
    emits: [scan_item_found, scan_complete, scan_item_failed]

  filter_manifest_by_path_type:
    description: "Filters manifest by path type to create image priority manifest"
    command: create_image_priority_manifest
    emits: [priority_manifest_created]

  begin_manifest_upload:
    description: "Actually does the uploading and records that upload started"
    command: upload_blob_using_manifest
    emits: [manifest_upload_started]

policies:
  update_partition_manifest:
    description: "Updates partition manifest when scan items are found"
    event: scan_item_found
    emits: []

  create_image_priority_manifest:
    description: "Issues command to create image priority manifest when scan completes"
    event: scan_complete
    emits: [create_image_priority_manifest]

  start_manifest_upload:
    description: "Unwraps manifest list and generates upload commands for each item"
    event: priority_manifest_created
    emits: [upload_blob_using_manifest]

Stage 1 — Parse

Structural lift from YAML. Enumerates all processes and channels. Channel types are unresolved (?). Validates graph integrity: every subscription has a publisher, no orphan emitters.

Stage 2 — Locate

Assigns a deployment location to each process (e.g. container/X). Today: one container per process. Future: observational data and preferences drive co-location decisions.

Stage 3 — Resolve verify here

Compares publisher and subscriber locations. Same location → direct. Cross-location → queue. This is the document a human signs off on before IaC is generated.

Channel type resolution rule

After location assignment, each channel's type is determined by comparing publisher and subscriber locations:

function resolveChannelType(pub, sub) {
  if (pub.location === sub.location)
    return 'direct'    // same container
  if (pub.host === sub.host)
    return 'unix-socket'
  return 'queue'       // cross-container
}

Future: observational data

A separate data source (metrics, throughput, cost) can inform whether two processes should share a location. Co-located processes get direct channels — no queue overhead. This is a deployment topology decision, not a code change.

Future — Closing the Loop

The pipeline today is a one-way transform from feat file to IaC. The vision is a feedback loop where observational data and stored preferences flow back into Stage 2 (location assignment), making deployment topology an adaptive, data-driven decision.
Now

Trivial location assignment

One process per container. All channels resolve to queue. The pipeline is deterministic and manual — a human reviews the Resolved IR and approves IaC generation.

Near

Preference store

A structured YAML or database of deployment preferences per process (preferred host, scaling tier, co-location rules). Stage 2 reads from this store rather than applying a blanket "one container" rule.

Later

Observational data → topology decisions

Runtime metrics (throughput, latency, queue depth) feed back into Stage 2. A process pair with high-frequency synchronous messaging may be co-located automatically, converting their shared channel from queue to direct. A cost reduction with zero code change.

Vision

The IR as the stable interface

The feat file describes what. The Resolved IR describes how for a given moment in time. IaC renderers (docker-compose, pulumi, k8s) are pure functions of the IR. The process functions themselves never change — only the topology of how they're connected adapts.

The Process Protocol is the same bet at every layer: separate the logic from the transport. At the function level: logic vs emit callbacks. At the deployment level: feat file vs channel type resolution. The IaC is just another transport.
